CAPA 01
El punto de partida

Máquina local y sistema operativo

Idea central

Todo programa se construye en algún sitio antes de llegar al usuario. Ese sitio es tu propia máquina: un portátil, con su sistema operativo, un editor donde escribir el código y un lugar donde guardar los archivos del proyecto. Suena obvio, pero es la primera fuente de fricción cuando alguien empieza.

Hay que distinguir dos cosas que se confunden: el sistema operativo (Windows, macOS, Linux) y el entorno de desarrollo (el editor y las herramientas que instalas encima). Muchas veces se instala una capa Linux dentro de Windows para que las herramientas modernas funcionen sin roces. Todas las capas siguientes se apoyan en esta.

Componentes
Sistema operativo Windows · macOS · Linux

El software base que gestiona el hardware y coordina el resto de programas. Los tres dominantes son Windows (Microsoft), macOS (Apple) y Linux (familia de sistemas abiertos).

Para desarrollar, macOS y Linux comparten una base común (Unix) que los hace muy amigos de las herramientas técnicas. Windows tradicionalmente ha sido menos amigable, pero desde hace años Microsoft ha resuelto eso con WSL.

Distribución de Linux Ubuntu · Debian · Fedora

Linux no es un producto único; es una familia de distribuciones (o "distros"), cada una con su gestión de paquetes y su comunidad. Ubuntu es la más popular para principiantes y se ha convertido en el estándar de facto en entornos de desarrollo y en servidores.

Cuando alguien dice "instalé Ubuntu", se refiere a la distribución. Cuando dice "en Linux", se refiere a la familia.

WSL Ubuntu dentro de Windows

Windows Subsystem for Linux. Una capa oficial de Microsoft que permite ejecutar Ubuntu (u otras distros) dentro de Windows sin la sobrecarga de una máquina virtual. Es lo que hoy usan la mayoría de desarrolladores en Windows para que las herramientas de línea de comandos se comporten como en un servidor Linux real.

Traducción práctica: puedes tener el Explorer de Windows para tus documentos y una terminal Ubuntu para trabajar con el código.

Editor de código y IDE VS Code · Cursor · JetBrains

Un editor de código es un programa donde escribes y editas archivos de código. Un IDE (Integrated Development Environment) es un editor más las herramientas adyacentes: depurador, terminal, control de versiones.

VS Code es el editor gratuito de Microsoft. Se ha convertido en el estándar de facto. Ligero, extensible, con soporte para prácticamente todos los lenguajes.

Existen variantes con IA integrada de fábrica —Cursor y Windsurf— y opciones profesionales de pago como los IDEs de JetBrains (WebStorm, PyCharm, IntelliJ).

Extensiones del editor marketplace

VS Code y equivalentes tienen un marketplace con miles de extensiones gratuitas que añaden capacidades: soporte para un lenguaje, integración con Git, IA de código, formateadores automáticos, temas visuales.

La mayoría de la fricción de arranque desaparece instalando cuatro o cinco extensiones básicas. La comunidad publica listas de recomendaciones para cada tipo de proyecto.

Sistema de archivos y rutas carpetas, paths y permisos

La manera en la que el SO organiza los archivos en disco. Cada archivo tiene una ruta que lo identifica de forma única (por ejemplo, /home/usuario/proyectos/mi-app/index.js).

En Linux y macOS los archivos tienen permisos explícitos (lectura, escritura, ejecución). En Windows también existen, pero suelen quedar más ocultos al usuario.

Toda esta capa es invisible en el día a día hasta que aparece un error de "permiso denegado" y se convierte en el primer obstáculo real que te encuentras.

En una reunión

Lo que oirás

"El equipo trabaja en macOS y en Windows con WSL Ubuntu; el editor estándar es VS Code con extensiones para Git y Copilot."

Traducción: uso mixto de sistemas operativos, con Ubuntu virtualizado dentro de los Windows para que las herramientas se comporten igual en ambas máquinas. Usan el mismo editor con las mismas extensiones para que el código se vea y se maneje igual, quien sea el que lo escriba.

CAPA 02
Hablar directo con la máquina

Línea de comandos

Idea central

Casi cualquier herramienta técnica pide, tarde o temprano, abrir una pantalla negra y escribir comandos. Esa pantalla es lo más antiguo del ordenador y, paradójicamente, sigue siendo el idioma nativo de los desarrolladores. Aunque suene retro, es donde el 90% de las tareas técnicas ocurren.

La buena noticia: no hace falta aprenderla entera. Con seis o siete comandos básicos y confianza para pegarle instrucciones que la IA te dicte, se avanza muchísimo. La fluidez con el terminal es probablemente la habilidad con mayor retorno por hora invertida en toda esta capa.

Componentes
CLI y GUI texto vs. ventanas

CLI (Command Line Interface): interfaz basada en texto. Escribes un comando, la máquina responde. Sin ratón, sin ventanas.

GUI (Graphical User Interface): interfaz visual con ventanas, iconos y botones. Es lo que usas cada día en Word, en un navegador o en el explorador de archivos.

Muchas tareas técnicas solo existen en CLI, o van mucho más rápido allí. Por eso la ganamos al principio, no al final.

Terminal y shell aplicación y motor

La terminal es la aplicación que abre la ventana negra. El shell es el programa que interpreta lo que escribes: recibe tu comando, lo ejecuta y te devuelve el resultado.

Los shells más comunes: PowerShell (nativo de Windows, potente para administración), Bash (tradicional de Unix/Linux, estándar en servidores) y Zsh (moderno, popular en macOS).

Prompt de terminal la señal de que espera

El símbolo (típicamente $ en Linux/macOS o > en Windows) que indica que el shell está listo para recibir tu siguiente instrucción. Suele ir precedido por información útil: usuario, máquina, carpeta actual, rama de Git.

No hay que confundirlo con el "prompt" de la IA. Ambos son "invitaciones a escribir", pero uno se dirige a la máquina y el otro a un modelo.

Anatomía de un comando comando + argumentos + flags

Un comando se compone típicamente de tres partes: el nombre del programa que quieres ejecutar, cero o más argumentos (sobre qué actúa) y opcionalmente flags u opciones (modificadores del comportamiento).

Ejemplo: en ls -l documentos, "ls" es el comando (listar), "-l" es un flag (formato de lista larga) y "documentos" es un argumento (qué listar).

Comandos básicos de navegación cd, ls, mkdir, cat

Los que aparecen en cualquier tutorial:

  • cd (change directory): cambia la carpeta en la que estás.
  • ls o dir: lista lo que hay en la carpeta actual.
  • mkdir: crea una carpeta nueva.
  • cat: muestra el contenido de un archivo por pantalla.
  • pwd: te dice en qué carpeta estás.

Con estos cinco, ya te mueves con soltura.

Directorio de trabajo la carpeta actual

El directorio de trabajo es la carpeta en la que el shell te sitúa por defecto. Todos los comandos que ejecutes se aplicarán ahí, salvo que especifiques otra ruta.

Cuando alguien dice "sitúate en la raíz del proyecto", está pidiendo que uses cd para llegar a la carpeta principal antes de la siguiente instrucción.

Variables de entorno configuración accesible al shell

Valores nombrados que existen para el shell y para los programas que arranca. Algunas son universales (como PATH, que dice dónde buscar los ejecutables), otras son propias de un proyecto (como una clave API o la URL de una base de datos).

Se guardan típicamente en archivos .env que no se suben a repositorios públicos por seguridad.

En una reunión

Lo que oirás

"Abre una terminal, sitúate en la raíz del proyecto y ejecuta el comando con los flags que te pasa la IA. Comprueba que las variables de entorno están cargadas."

Traducción: abre el shell, ve a la carpeta principal del código, pega el comando que la IA te haya dicho tal cual, y asegúrate de que el archivo con las claves de configuración está en su sitio. Ninguna de estas cuatro cosas requiere saber programar; requiere haber usado la terminal media hora antes.

CAPA 03
Traer código ajeno

Runtimes y gestores de paquetes

Idea central

Nadie construye software moderno partiendo del vacío. Se combinan piezas que otros ya escribieron y compartieron abiertamente: enviar correos, autenticar con Google, procesar Excel, generar PDF. Cada una de esas piezas es un paquete.

Necesitas dos cosas para vivir en ese mundo: un runtime (el motor que hace correr esas piezas) y un gestor de paquetes (la biblioteca desde donde las descargas). Node.js con npm es el binomio más común; Python con pip es el equivalente en su ecosistema.

Componentes
Runtime el motor de ejecución

Un runtime es el programa que ejecuta código escrito en un lenguaje concreto. Node.js es el runtime que permite ejecutar JavaScript fuera del navegador, en tu portátil o en un servidor.

Alternativas modernas a Node: Deno y Bun, con foco en seguridad y velocidad. En el mundo Python, el runtime es simplemente "el intérprete de Python".

Gestor de paquetes npm · yarn · pnpm · pip

Herramienta que instala, actualiza y organiza las dependencias de tu proyecto. En Node.js se usan npm (viene por defecto), yarn o pnpm. En Python el estándar es pip.

Todos hacen lo mismo esencial: bajan código ajeno desde un registro público y lo dejan listo para que tu proyecto lo use.

Paquete y dependencia unidad de código reutilizable

Un paquete es una unidad de código publicada por terceros, con nombre, versión y documentación. Una dependencia es un paquete que tu proyecto necesita.

Un proyecto sencillo puede tener 20 dependencias directas y, a través de ellas, tirar de cientos de dependencias transitivas (dependencias de tus dependencias). Todo eso se instala automáticamente.

package.json y requirements.txt el manifiesto del proyecto

En Node.js, package.json es el archivo central del proyecto: nombre, versión, dependencias, scripts. En Python el equivalente es requirements.txt (o modernamente pyproject.toml).

Cuando alguien te comparte un proyecto, con un npm install o pip install -r requirements.txt reconstruyes todo su entorno.

Lockfile versiones congeladas

Archivo generado automáticamente que guarda las versiones exactas de todas las dependencias, directas y transitivas. Ejemplos: package-lock.json (npm), yarn.lock, pnpm-lock.yaml.

Sirve para reproducibilidad: si tú y otro colega instaláis el proyecto la misma semana, ambos obtenéis exactamente las mismas versiones. Debe subirse al repositorio.

Versión semántica (semver) 1.2.3 significa algo

Convención universal para numerar versiones de un paquete: MAYOR.MENOR.PARCHE. Un cambio de "parche" (1.2.3 → 1.2.4) arregla bugs sin cambiar el comportamiento. Uno "menor" (1.2 → 1.3) añade cosas sin romper. Uno "mayor" (1.x → 2.0) rompe la compatibilidad.

En la práctica sirve para saber si actualizar puede romper algo o no.

Registro público de paquetes npm · PyPI

Servidor público desde el que se descargan los paquetes. El npm registry aloja millones de paquetes JavaScript. PyPI hace lo mismo para Python. Son bienes comunes de la industria; casi todos los paquetes que instalas vienen de ellos.

En una reunión

Lo que oirás

"Instálalo con npm install, mira el package.json para ver qué dependencias tira, y no olvides commitear el lockfile."

Traducción: usa el gestor de paquetes para bajar todas las librerías, revisa el archivo descriptor del proyecto para ver qué está usando, y asegúrate de guardar en el repositorio el archivo que congela las versiones exactas para que otros puedan reproducir tu entorno.

CAPA 04
El alfabeto que la máquina obedece

Lenguajes de programación

Idea central

Un lenguaje de programación es un conjunto de reglas para expresar instrucciones que la máquina puede ejecutar. Hay decenas, y cada uno tiene una comunidad, un dominio dominante y una curva de aprendizaje distintos.

Para el producto comercial (webs, dashboards, integraciones con CRM) hay tres nombres que aparecen sin falta: JavaScript y su versión mejorada TypeScript en el frontend, y Python en el backend y análisis de datos. Con esos tres se cubre el 90% de lo que probablemente necesites.

Componentes
Compilado vs. interpretado dos maneras de ejecutar

Un lenguaje compilado traduce todo el código a un archivo ejecutable antes de correrlo. Rápido en producción, más rígido para iterar. Ejemplos: Go, Rust, C.

Un lenguaje interpretado se ejecuta línea a línea por un runtime. Más flexible y rápido de iterar, algo más lento en ejecución. Ejemplos: JavaScript, Python.

Es distinción académica pero se filtra en las conversaciones cuando se habla de rendimiento.

Tipado estático vs. dinámico chequeo antes o después

En tipado estático el tipo de cada variable (número, texto, lista) se declara y se comprueba antes de ejecutar. Detecta errores temprano; obliga a más disciplina. Ejemplos: TypeScript, Go.

En tipado dinámico los tipos se comprueban al ejecutar. Más flexible, más rápido de escribir, más fácil de romper. Ejemplos: JavaScript puro, Python.

Para proyectos serios, el tipado estático suele ganar. Para prototipos rápidos, el dinámico.

JavaScript el lenguaje universal de la web

El único lenguaje que ejecutan todos los navegadores del mundo. Nació para pequeños scripts de página, hoy corre desde la web hasta servidores (Node.js), aplicaciones móviles (React Native) y servidores de escritorio (Electron).

Es imposible construir para la web sin tocar, aunque sea de refilón, JavaScript.

TypeScript JavaScript con red de seguridad

Extensión de JavaScript creada por Microsoft que añade tipado estático. En la práctica: te avisa mientras escribes de errores que en JavaScript solo verías al ejecutar. Es el estándar en producto empresarial serio y el favorito de la IA porque le da más pistas.

Se escribe casi igual que JavaScript. Antes de ejecutarse se traduce ("transpila") a JavaScript normal.

Python datos, IA y automatización

Lenguaje interpretado con sintaxis muy legible ("casi como leer inglés"). Domina en tres dominios: análisis de datos, IA/machine learning, y automatización de procesos (scripts, ETL, integraciones).

Si tu herramienta va a leer Excel, cruzar CSV con bases de datos o llamar a modelos de IA, Python es probablemente la elección natural.

Otros lenguajes que oirás Go, Rust, Java, C#

Go: lenguaje compilado de Google. Usado en infraestructura y microservicios.

Rust: compilado, foco en rendimiento y seguridad de memoria. Emergente.

Java: histórico, base de sistemas empresariales. Corre en cualquier sitio.

C#: el equivalente de Microsoft, base del ecosistema .NET. Muy común en corporaciones que ya viven en Windows.

No los necesitas para tu primer proyecto, pero aparecerán en conversaciones con clientes.

HTML y CSS no son de programación, pero conviene saberlo

HTML es un lenguaje de marcado: describe qué elementos hay en una página (títulos, párrafos, botones, imágenes). CSS es un lenguaje de estilos: dice cómo se ven esos elementos (colores, tamaños, disposición).

Ninguno es de programación en sentido estricto, pero cualquier interfaz web se compone de HTML + CSS + JavaScript. Los frameworks modernos (Capa 05) los generan por ti, pero el nombre aparece constantemente.

Linter y formateador buen estilo automatizado

Un linter analiza tu código y avisa de errores obvios y de estilos malos antes de ejecutarlo (ESLint para JS/TS, Ruff para Python). Un formateador reordena y reindenta el código para que siga un estilo consistente (Prettier para JS/TS, Black para Python).

Los dos se ejecutan automáticamente al guardar y te ahorran discusiones estéticas eternas.

En una reunión

Lo que oirás

"El backend en Python con FastAPI, el frontend en TypeScript con React. Prettier y ESLint activos, con Ruff en el lado Python."

Traducción: dos lenguajes, uno en cada lado del producto. Utilizan formateadores y linters que aseguran que el código de todo el equipo se ve y comporta igual. Es una elección estándar y madura; no requiere debate técnico.

CAPA 05
Materiales elegidos con propósito

Frameworks y stack técnico

Idea central

Sobre los lenguajes se apoyan los frameworks: paquetes grandes de código pre-hecho que resuelven problemas comunes —renderizar interfaces, gestionar rutas web, conectarse a bases de datos, autenticar usuarios—. Sin framework, tendrías que reinventar todo eso cada vez.

Elegir un stack —la combinación de lenguaje + framework + herramientas— es la primera decisión estratégica de cualquier proyecto. La buena noticia: para casos comerciales comunes, existen combinaciones estándar que la industria ya usa. No necesitas inventar; necesitas reconocer.

Componentes
Framework vs. librería quién llama a quién

Una librería es un conjunto de funciones que tú llamas cuando quieres. Un framework es una estructura completa: tú escribes tu código dentro de sus reglas, y él te llama a ti cuando corresponde.

Metafóricamente: una librería es una caja de herramientas; un framework es un taller ya montado con normas de trabajo.

Frontend, backend y full stack las dos mitades del producto

Frontend: la parte que ve el usuario en el navegador o la app. Botones, formularios, gráficas.

Backend: la parte que corre en un servidor, invisible para el usuario. Lógica de negocio, acceso a base de datos, APIs.

Full stack: perfil o proyecto que abarca las dos. Muchos proyectos pequeños son full stack por definición.

React y Next.js el estándar del frontend moderno

React es una librería de JavaScript (creada por Facebook) para construir interfaces con componentes reutilizables. Es dominante en producto empresarial.

Next.js es un framework que envuelve React y le añade lo que le falta: sistema de rutas, renderizado en servidor, optimizaciones, despliegue integrado con Vercel. En 2026 es la base habitual de webs de producto.

Alternativas: Vue, Svelte, Astro otras familias del frontend

Vue: alternativa a React con enfoque más accesible. Muy popular en Asia y Europa.

Svelte: framework moderno que compila a JavaScript ultraligero. Buen rendimiento con poco código.

Astro: enfocado a webs orientadas a contenido (blogs, documentación), con muy poca sobrecarga en el navegador.

Todas son elecciones legítimas. Para una primera herramienta, la comunidad que hay detrás de React/Next es lo que suele decantar la balanza.

Tailwind CSS estilos por composición

Sistema de estilos que aporta cientos de clases utilitarias pequeñas (padding, colores, tipografías, layouts) que se combinan directamente en el HTML. Elimina la necesidad de escribir CSS a mano y produce interfaces coherentes rápido.

Se ha convertido en el compañero habitual de React y Next.js en producción.

Backends comunes Express, FastAPI, Django

Express: framework mínimo para backends en Node.js. Rápido y sin opiniones fuertes.

FastAPI: framework moderno de Python especializado en APIs. Muy rápido, con tipado.

Django: framework veterano de Python, "batteries included" (base de datos, autenticación, panel de admin fuera de la caja).

Para una primera herramienta con IA, FastAPI es una elección muy natural.

Boilerplate y estructura el esqueleto de un proyecto

Boilerplate: código repetitivo mínimo necesario para arrancar un proyecto. Muchas plantillas y comandos (create-next-app, npm create vite) lo generan por ti en segundos.

La estructura de carpetas típica separa la interfaz (/src/components, /src/pages), la lógica (/src/lib) y los recursos estáticos (/public).

Componente y estado los átomos del frontend

Un componente es un bloque reutilizable de interfaz: un botón, una tarjeta, un formulario. Los componentes se combinan como piezas de lego para construir pantallas.

El estado son los datos que la aplicación mantiene en memoria y que cambian con la interacción (qué texto has escrito, qué opción has seleccionado). Gestionar bien el estado es la mitad del trabajo de un frontend.

MVP y arquitectura dimensionada no construir una catedral

Un MVP (Minimum Viable Product) es una versión mínima de un producto que ya aporta valor. La disciplina de dimensionar bien: si tu herramienta solo va a usarla tu equipo de 5 personas, no necesita la arquitectura que sostiene a Amazon.

Este es probablemente el punto más importante de toda la capa: la mayoría de proyectos fallan por sobreingeniería, no por falta de recursos.

En una reunión

Lo que oirás

"Vamos a levantar el MVP con Next.js y Tailwind, backend en FastAPI y una base de datos ligera. Estructura por componentes y sin sobreingeniería."

Traducción: eligen un stack estándar, ampliamente adoptado, para poner en marcha una versión mínima funcional. Están anunciando que no van a construir una catedral: van a resolver el caso de uso con lo justo. Es una respuesta pragmática y madura.

CAPA 06
El aprendiz que también ejecuta

IA como copiloto de desarrollo

Idea central

La IA ha cambiado el desarrollo. No sustituye al programador experto, pero acelera brutalmente al que sabe pedirle bien. Y —esta es la puerta que abre este volumen— hace posible que alguien sin formación técnica formal construya herramientas útiles a partir de conversaciones bien dirigidas con un asistente.

La clave está menos en escribir código que en saber describir problemas: qué quieres que haga la herramienta, qué debe pasar cuando algo no funciona, qué es aceptable y qué no. La capa lingüística sube de nivel; la capa técnica baja.

Componentes
LLMs generalistas ChatGPT · Claude · Gemini

Los modelos de lenguaje grandes son la base. ChatGPT (OpenAI) fue el primer asistente popular en mainstream. Claude (Anthropic) destaca por su calidad en tareas de razonamiento largo y código. Gemini (Google) se integra con su ecosistema.

Cualquiera de los tres, bien pedido, produce código funcional para tareas comerciales típicas.

Autocompletado en el editor GitHub Copilot

GitHub Copilot fue el primer producto masivo: sugerencias de código en línea mientras escribes, aceptables con la tecla Tab. Funciona sobre VS Code y otros editores.

No sustituye la conversación; complementa. Es útil especialmente cuando ya sabes lo que quieres escribir y solo necesitas agilidad.

Editores agenticos Cursor · Windsurf · Claude Code

La generación siguiente: editores que no solo autocompletan, sino que leen tu proyecto entero, entienden el contexto y son capaces de modificar varios archivos a la vez a partir de una instrucción en lenguaje natural.

  • Cursor: variante de VS Code con IA integrada. Muy popular.
  • Windsurf: equivalente, con enfoque en flujos agenticos.
  • Claude Code: herramienta de línea de comandos de Anthropic para delegar tareas de código.
  • Aider: cliente open-source similar, desde la terminal.

Son el punto donde alguien sin formación técnica puede construir productivamente.

Prompt y contexto las dos palancas

El prompt es tu instrucción a la IA. Su calidad determina la calidad del resultado. Un buen prompt describe el qué (objetivo), el cómo (estilo, restricciones) y el cuándo funcionaría (criterio de éxito).

El contexto es todo lo demás que la IA tiene a mano al responder: archivos abiertos, mensajes previos, documentación adjunta. Un prompt vago con buen contexto puede funcionar mejor que un prompt largo sin contexto.

Referencia de archivos @archivo, /add

Casi todas las herramientas modernas permiten adjuntar archivos concretos al contexto del asistente (@nombre-archivo, /add). Es la manera más eficaz de decirle a la IA sobre qué queremos que actúe.

Regla práctica: mejor tres archivos bien seleccionados que veinte irrelevantes.

Vibe coding y desarrollo dirigido dos modos legítimos

Vibe coding: programar guiado por intuición, iterando con la IA sin planificación previa. Muy útil para prototipar rápido y descubrir qué queremos.

Desarrollo dirigido: planificar la estructura, escribir prompts claros, revisar cada cambio. Necesario cuando el código va a producción.

Los dos modos son legítimos si sabes en cuál estás. El error es hacer vibe coding y creer que ya está listo para producción.

Riesgos que hay que conocer alucinaciones, dependencia oscura, bugs sutiles

La IA a veces inventa funciones o librerías que no existen (alucinaciones de código). Puede introducir paquetes desconocidos como dependencia (dependencia oscura). Puede resolver un caso obvio y fallar en uno raro (bugs sutiles).

Contramedidas: revisar cada cambio grande, ejecutar el código en local antes de aprobarlo, mantener un buen conjunto de datos de prueba y —si el proyecto lo justifica— pedir a un desarrollador humano una revisión antes de publicarlo.

En una reunión

Lo que oirás

"Prototipé la landing con Cursor y Claude, adjunté los tres archivos relevantes al contexto, y revisamos el diff antes de commitear."

Traducción: usaron un editor con IA para escribir el primer código, le pasaron los archivos del proyecto para que la IA entendiera el contexto, y aun así revisaron manualmente los cambios antes de guardarlos formalmente. Es un flujo de trabajo maduro con IA; no es magia, es dirección.

CAPA 07
Cobrar vida en tu propia máquina

Ejecución local

Idea central

Antes de compartirlo con el mundo, tu programa se prueba en tu propia máquina. Enciendes un servidor de desarrollo, tu portátil se convierte por un momento en un pequeño servidor web, y abres el resultado en una dirección peculiar: localhost:3000.

Este ciclo entre escribir código, ver el cambio en el navegador en segundos y volver al editor es la esencia del desarrollo moderno. La productividad de todo lo que hagas después depende de que este ciclo esté bien engrasado.

Componentes
Servidor de desarrollo tu portátil como servidor

Un proceso que corre en tu propia máquina y sirve tu aplicación mientras la desarrollas. Se arranca con un comando (típicamente npm run dev) y queda vivo hasta que lo paras.

Ese servidor es solo tuyo: nadie de fuera puede acceder. Es el entorno de pruebas privado donde iteras a máxima velocidad.

localhost y puertos tu propia dirección

localhost es el nombre reservado que apunta a tu propio ordenador. Cuando el servidor de desarrollo se arranca, tu navegador puede visitarlo escribiendo localhost:3000.

Ese 3000 es el puerto: un número que identifica un servicio dentro de una máquina. Los frameworks suelen usar puertos por defecto (3000 en Next.js, 5173 en Vite, 8000 en FastAPI). Si el puerto ya está ocupado, saltará automáticamente al siguiente.

Script npm atajos definidos en package.json

En package.json se declaran comandos con nombre: dev, build, test, lint. Se ejecutan con npm run <nombre>. Sirven para no tener que recordar comandos largos: npm run dev arranca el servidor de desarrollo tal y como el proyecto está configurado.

Cuando abres un proyecto nuevo, mirar los scripts definidos es la mejor manera de saber qué puedes hacer con él.

Compilación (build) de desarrollo a producción

La compilación o build es el proceso de traducir el código fuente a una versión optimizada, comprimida y lista para producción. Los frameworks modernos lo hacen automáticamente con un solo comando.

Producto lateral: mientras el modo desarrollo prioriza velocidad de iteración, el build prioriza tamaño y velocidad de carga final. Nunca se sirve al usuario el código de desarrollo tal cual.

Bundler Vite · Webpack · esbuild

Un bundler es la herramienta que toma decenas de archivos de código y los junta en unos pocos optimizados. Es la pieza que hace posible el paso de "diez mil líneas repartidas" a "una web que carga en 200 ms".

Vite es el bundler moderno y muy rápido, estándar en proyectos recientes. Webpack es el histórico, muy configurable pero lento. esbuild es el motor ultrarrápido detrás de muchos otros.

Hot reload (HMR) cambia y ve el cambio al instante

Hot Module Replacement. Cuando cambias un archivo del proyecto, el servidor de desarrollo recarga solo esa parte en el navegador sin reiniciar todo. En Vite o Next.js, el cambio suele reflejarse en menos de un segundo.

Esta capacidad, que hace 10 años era exótica, es hoy la base de la productividad de cualquier proyecto frontend.

DevTools del navegador la ventana al interior

Herramientas incorporadas en cualquier navegador moderno (F12 en la mayoría). Permiten inspeccionar el HTML renderizado, los estilos aplicados, las peticiones de red, los mensajes de la consola, y el rendimiento.

Son la primera parada cuando algo no funciona como esperas. La IA suele pedirte "abre la consola y dime qué ves".

Variables de entorno (.env) configuración local, no versionada

Un archivo .env guarda valores sensibles o dependientes del entorno: claves API, URLs de bases de datos, credenciales. Cada desarrollador tiene el suyo local; nunca se sube al repositorio (por eso .env aparece siempre en .gitignore).

Es el primer sitio donde miran los desarrolladores cuando algo no arranca. Es también uno de los orígenes de incidentes de seguridad más frecuentes cuando se sube por error.

Depuración: logs, breakpoints, stack traces encontrar por qué falla

Cuando algo no funciona, tienes tres armas: logs (mensajes que la aplicación escribe describiendo lo que hace), breakpoints (puntos donde el código se para y puedes inspeccionar todo) y stack traces (el rastro del error a través de las funciones que lo produjeron).

La IA es muy buena leyendo stack traces. Pásaselo tal cual: te dará una hipótesis en segundos.

En una reunión

Lo que oirás

"Arranca el dev server con npm run dev en localhost:3000, verás el hot reload a cada cambio. Si algo peta, mira la consola y pásale el stack trace a Claude."

Traducción: enciende el entorno local de pruebas en tu propia máquina. Cuando cambies el código lo verás reflejado al instante. Si algo falla, abre el panel del navegador donde salen los errores y copia el mensaje completo al asistente de IA para diagnosticarlo. Es el ciclo estándar de trabajo diario.

CAPA 08
De tu portátil al mundo

Repositorio y despliegue público

Idea central

El último salto: sacar el código de tu portátil y ponerlo en un lugar donde otros puedan usarlo. Son dos movimientos combinados. Primero, guardar el código en un repositorio remoto (GitHub) para que no dependa de tu disco duro. Segundo, publicarlo en una plataforma de despliegue (Railway, Vercel, Netlify) que le dará una URL pública accesible desde cualquier sitio.

La belleza del ecosistema actual: bien configurado, cada vez que guardas un cambio en el repositorio, la plataforma lo despliega automáticamente. El flujo de "cambio local → producción" puede llegar a durar menos de un minuto.

Componentes
Git y control de versiones memoria completa del proyecto

Git es un sistema que registra cada cambio del código con quién lo hizo, cuándo y por qué. Permite volver atrás, comparar versiones, colaborar sin pisarse. Es la base invisible de todo el desarrollo profesional moderno.

Se usa desde la terminal o desde el propio VS Code. La IA lo maneja bien; en la práctica, con tres comandos (add, commit, push) haces el 90% del uso diario.

Repositorio (GitHub, GitLab, Bitbucket) el archivo en la nube

Un repositorio es el contenedor del código con su historial completo. GitHub es la plataforma dominante que lo aloja en la nube: es simultáneamente archivo, red social de desarrolladores y sistema de colaboración.

GitLab y Bitbucket son alternativas equivalentes, populares en entornos corporativos.

Commit, branch, merge el vocabulario de Git

Commit: confirmar un conjunto de cambios en el historial, con un mensaje descriptivo.

Branch o rama: copia paralela del código donde trabajar sin afectar la línea principal. Se usa para probar cosas o desarrollar nuevas funcionalidades.

Merge: integrar los cambios de una rama en otra.

La rama principal se llama tradicionalmente main (antes master). Un flujo típico: crear una rama, hacer cambios, commitear, abrir un pull request, revisar, mergear a main.

Push, pull y clone sincronizar con la nube

Clone: descargar por primera vez un repositorio remoto para tener una copia local.

Push: enviar tus commits locales al repositorio remoto.

Pull: descargar los cambios remotos que otros hayan hecho.

Con estos tres verbos —clone, push, pull— más el trío add/commit/push del punto anterior, tienes la operativa diaria completa.

.gitignore y README qué ignorar, cómo presentarse

.gitignore: archivo que lista qué carpetas y ficheros Git debe ignorar. Típicamente: node_modules, archivos .env con credenciales, carpetas de build.

README: documento de presentación del repositorio (formato Markdown). Describe qué es el proyecto, cómo instalarlo y cómo usarlo. Es lo primero que verá cualquier persona (o IA) que aterrice en tu repositorio.

Deploy (despliegue) de código a servicio activo

El deploy es el proceso por el que tu código pasa a estar corriendo en un servidor real, accesible por internet. Antaño requería configurar servidores manualmente; hoy plataformas especializadas lo automatizan por completo.

El patrón moderno: conectas la plataforma a tu repositorio de GitHub. Cada vez que haces push a la rama principal, la plataforma reconstruye y despliega la nueva versión automáticamente.

Plataformas de despliegue Vercel · Netlify · Railway · Render

Vercel: especializada en frontends modernos (Next.js). Muy pulida, gratuita para proyectos pequeños.

Netlify: equivalente a Vercel, muy usada para webs estáticas y frontends.

Railway: enfocada a aplicaciones con backend y bases de datos. Muy popular en prototipos con Python o Node.

Cloudflare Pages: opción gratuita generosa, apoyada en la red global de Cloudflare.

Render: alternativa versátil para apps con backend y datos persistentes.

Build automático y CI/CD el ciclo desatendido

Las plataformas modernas ejecutan un build automático cada vez que detectan un push: descargan el código, instalan las dependencias, compilan y despliegan. Todo esto sin intervención humana.

El concepto general se llama CI/CD (Continuous Integration / Continuous Deployment). En un proyecto pequeño está ya montado por defecto; en uno grande se personaliza mucho.

URL, dominio y subdominio la dirección que compartes

La URL pública es la dirección accesible desde internet. Al principio será una URL genérica que te asigna la plataforma (mi-herramienta.vercel.app).

Después puedes conectarla a un dominio propio (brompton.com) o a un subdominio (leads.brompton.com). Los dominios se compran a un registrador (Namecheap, Cloudflare, Google Domains) por precios modestos.

Rollback volver atrás cuando algo falla

Todas las plataformas modernas permiten hacer rollback instantáneo: volver a una versión anterior con un solo clic si el último deploy rompe algo. Combinado con el historial de Git, hace que trabajar con miedo a "romperlo todo" sea absurdo.

Este es el punto donde la infraestructura moderna deja de ser un lujo y se convierte en una red de seguridad para quien empieza.

En una reunión

Lo que oirás

"El repo está en GitHub con deploy automático a Railway; cada push a main dispara build y despliegue en la URL pública. Si algo falla, rollback en un clic."

Traducción: el código está guardado en la nube. Cada cambio confirmado en la rama principal genera automáticamente una versión nueva accesible por internet. Y si esa versión rompe algo, se vuelve atrás al instante. Es el flujo de trabajo profesional estándar en 2026 —y está al alcance sin escribir una sola línea de configuración manual.
